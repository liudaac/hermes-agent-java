多租户 AI 组织运行时核心说明

本文档梳理 Hermes Agent Java 当前整体架构、核心模块关系、AI 组织原生化“五刀”闭环、端到端 Release 演练，以及后续演进路线。

1. 项目定位

当前 Hermes Agent Java 已经不再是单纯的 Agent Runtime，而是在往 多租户 AI 组织操作系统 演进。

一句话定位：AI-native multi-tenant agent organization runtime —— 多租户 AI 组织运行时。

过去的主链路更像：

用户输入 → Agent 调模型 → 调工具 → 返回答案

现在的主链路已经变成：

用户/平台请求
  → 多租户运行时
  → Agent 组织成员
  → 团队协作
  → 工具执行
  → 记忆/进化/可观测
  → Dashboard/Gateway 暴露能力

核心变化是：Agent 不再只是一个孤立工具使用者，而是有身份、有角色、有团队、有知识库、有失败教训、有组织追踪的“组织成员”。

4. Agent 核心层

AIAgent

原始单体 Agent 核心，主要负责模型调用、对话历史、system prompt、工具调用、memory、skills、approval、trajectory、reflection 等。

TenantAwareAIAgent

当前真正的主角。在 AIAgent 基础上加入多租户、角色、团队、总线、自进化、可观测等组织原生能力。

TenantAwareAIAgent 的职责

TenantAwareAIAgent
├── 绑定 tenantId / TenantContext
├── 使用租户级 MemoryManager
├── 使用租户级 ToolDispatcher
├── 自动注册 AgentRole
├── 自动加入 Team
├── 自动注册 TenantBus
├── 构造组织感知 system prompt
├── 注入 team context
├── 注入 evolution context
├── 处理工具调用
├── 记录 tool success/failure
├── 记录 AgentTrace
└── 处理 bus message

因此，TenantAwareAIAgent 现在可以看成一个属于某个租户、拥有角色、能加入团队、能收发消息、能学习、能被追踪的组织成员。

5. 工具系统

工具层由三个核心组成：

ToolRegistry

全局工具注册表，保存工具名、toolset、schema、handler、risk、approval config。

ToolInitializerV2

启动时注册全部工具，包括新增的 OrgNativeTools。

TenantAwareToolDispatcher

工具执行的租户边界。把文件、代码、memory、org 工具路由到对应租户组件。

工具执行路由

tool call
  ↓
TenantAwareToolDispatcher
  ↓
按类型路由：
  ├── 文件工具 → TenantFileSandbox
  ├── 代码/终端 → Tenant sandbox
  ├── memory 工具 → TenantMemoryManager
  ├── org 工具 → TenantContext 里的组织组件
  └── generic tool → ToolRegistry

7. collaboration/*：组织运转机制

collaboration/ 是更“活”的组织协作层，它让 Agent 之间形成队伍、通信、分工、协商和治理。

AgentRole

定义 Agent 在组织中的角色、级别、技能、可用工具和协作者。

new AgentRole("release-manager", "Deploys releases", AgentRole.Level.LEAD)
  .skills("release", "deploy", "production", "ci-cd");

TenantBus

Agent 间消息总线，支持 REQUEST、RESPONSE、NOTIFY、PROPOSE、ESCALATE、BROADCAST。

Team / TeamManager

团队抽象。包含 mission、members、lead、sharedState、recentActivity。

IntentOrchestrator

从自然语言 intent 出发，自动分解任务、匹配队友、执行并在失败时重派。

TaskOrchestrator

显式任务/流程编排，更接近 DAG 或 pipeline。

Negotiator / GovernancePolicy / OrgHealthChecker

协商、治理策略和组织健康检查。

8. AI 组织原生化“五刀”闭环

五刀解决了之前“有模块但不像活组织”的问题。

刀主题关键能力代表工具/类
第一刀Org-Native Tools让 Agent 感知组织：找队友、委派、查知识、升级人类OrgNativeTools
第二刀SelfEvolution 回路失败 → 根因 → 经验 → 下次 prompt 注入SelfEvolutionEngine
第三刀Team 共享上下文团队、成员、共享决策、团队 activityTeam, TeamManager
第四刀Intent 自我组织任务分解、技能匹配、自动委派、失败重派IntentOrchestrator
第五刀Org-wide 可观测tool trace、异常检测、行为回放AgentObservability, AgentTrace

9. 一次请求的完整执行流

1
用户/平台消息进入
来自 Gateway、Dashboard、ACP 或 Web UI。
2
定位 session 与 tenant
SessionManager / TenantManager 找到对应租户上下文。
3
TenantAwareAIAgent.processMessage()
检查 tenant 状态、quota、activity，并开启 AgentTrace。
4
buildSystemPrompt()
注入默认身份、memory、tools、skills、evolution context、team context。
5
ModelClient 调模型
模型返回自然语言或 tool calls。
6
executeToolCall()
角色权限检查、记录 TOOL_CALL、通过 TenantAwareToolDispatcher 执行、记录 TOOL_RESULT/ERROR。
7
进化与观测
成功 recordSuccess，失败 recordFailure，最终 completeTrace。
8
返回结果
结果回到 Gateway / Dashboard / 用户。

14. 当前强项与薄点

强项

多租户底座完整

Sandbox、quota、security、memory、skill、session、audit 都已具备。

Agent 不再孤立

role、team、bus、intent、trace 让 Agent 开始像组织成员。

组织闭环成型

知识 → 分工 → 执行 → 失败 → 学习 → 追踪 → 回放。

测试覆盖扎实

五刀新增 80 个测试，并有 Release E2E 演练。

仍需打磨的地方

薄点原因建议
IntentOrchestrator 仍偏启发式目前靠 token / skill / role / level 打分升级为 LLM + embedding + trace-informed planner
team tools 缺精确 agentId 上下文Dispatcher 当前主要持有 tenantContext让 TenantAwareToolDispatcher 感知当前 agentId
Dashboard 尚未完全吃下五刀数据后端能力已强，UI 可视化还可增强补 Team、Intent Run、Trace Timeline、Evolution Lessons 页面
Workflow 与 Intent 尚未融合一个显式 DAG，一个自组织计划让 IntentOrchestrator 生成 Workflow，由 WorkflowEngine 持久化执行

15. 推荐演进路线

1
UI 可视化五刀
Team、Intent Runs、Traces、Evolution Lessons、Anomalies、Tool Usage Heatmap。
2
IntentOrchestrator 智能化
从 heuristic 升级为 LLM / embedding / trace-informed planner。
3
融合 WorkflowEngine
intent → plan → workflow → execute → trace → evolve。
4
组织调度脑
引入 load、availability、failure rate、success pattern、cost 参与调度。
最终方向：从“多 Agent 框架”演进为真正的 AI 组织操作系统。
